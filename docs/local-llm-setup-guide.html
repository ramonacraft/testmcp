1. Why I did this

I built TestMCP — an MCP server (6 tools, 17 mapped functional areas) that analyzes GitHub pull requests and predicts prioritized QA test plans. I wanted two things: to never be locked into a single closed AI vendor, and to be able to demo my product running fully offline — no API keys, no cloud, wifi off.

In the current market, "model-agnostic" is what enterprise security reviews ask for. A QA platform that runs against Claude, GPT, or an air-gapped local model checks boxes that kill deals when missing. And for me personally, it upgrades the story from "I use AI tools" to "I built an AI product that runs on any model, including air-gapped."

2. Concepts: ownership, security, and what a local LLM is (and isn't)

Do you own it?

You own your copy. Open-weight models (Llama, Qwen, Mistral) are files you download to your disk. Nobody can revoke access, change them remotely, or meter your usage. You don't own the intellectual property — Meta/Alibaba license the models, mostly free for personal and commercial use — but functionally, the model works offline forever.

Is it secure?

Yes — that's the main selling point. Prompts, documents, and outputs never leave your machine. Nothing is sent to any company's servers. The one caution: only download models from the official sources (LM Studio's built-in catalog, Ollama's library, Hugging Face) — never random websites.

What it is NOT

An LLM is a brain, not a server. It doesn't host websites or serve files. Anything the public visits should live on a proper hosting service (GitHub Pages, Netlify, Vercel — all with free tiers). Exposing your personal computer to the internet is a security risk not worth taking. You can build apps that use your local LLM privately, but public hosting is a separate job.

3. Hardware reality check

Unified memory determines what you can run. Apple Silicon shares RAM with the GPU, which makes Macs surprisingly good at this.

RAMModel size that runs wellQuality expectation
8 GB3B comfortably; 7–8B (Q4) if you close other appsDrafting, summarizing, Q&A, tool calls (7B+)
16 GB13–14BNoticeably better reasoning
32 GB+30B+Approaches older GPT-4-class quality
Lesson: On 8 GB, skip anything 13B or larger — it swaps to disk and crawls. And for MCP/tool-calling work specifically, 7B is the floor: 3B models fumble structured tool calls.

4. Attempt 1: Ollama (and why I removed it)

My first install was Ollama — free, open source, terminal-based:

ollama run llama3.2

It worked great as a chat and as a local OpenAI-compatible API (http://localhost:11434). But my real goal was running TestMCP against a local model, and Ollama alone is not an MCP client — it can't drive MCP tools by itself.

Since LM Studio covers everything Ollama did (chat, local API server) plus acts as an MCP client, keeping both meant duplicate apps and duplicate multi-GB model files on a small disk. I removed Ollama:

# 1. Menu bar llama icon → Quit Ollama
# 2. Drag Ollama from Applications to Trash
# 3. Delete the hidden model folder (frees ~2 GB):
rm -rf ~/.ollama

# Verify it's gone — should say "No such file or directory":
ls ~/.ollama
Lesson: rm -rf prints nothing on success — silence IS the confirmation. Verify with ls if unsure. Also: pick your end goal first. If I'd known I needed an MCP client, I'd have started with LM Studio and saved a download.

5. The setup that stuck: LM Studio + Qwen2.5 7B

  1. Install LM Studio — download from lmstudio.ai, drag to Applications, launch. Free, no account needed for local use.
  2. Download a tool-capable model — in LM Studio's search tab, get Qwen2.5 7B Instruct, Q4 quantization (~4.5 GB). Tool-calling quality is what drives MCP servers, and this is the sweet spot for 8 GB.
  3. Load the model and chat to confirm it works. Close heavy apps (Chrome tabs especially) — a 7B model uses ~5 GB.

LM Studio also runs a local OpenAI-compatible API server (Developer tab → start server) — so anything you'd have scripted against Ollama works here too.

6. Wiring in TestMCP (the zero-code-change surprise)

Key insight: TestMCP never calls an LLM itself. It's an MCP server — the model always lives on the client side, and the risk engine is rule-based. That means it was already 100% model-agnostic by architecture. Zero code changes needed to swap Claude for a local model.

LM Studio supports MCP servers natively (v0.3.17+). Register TestMCP:

  1. In LM Studio: Program tab → Install → Edit mcp.json (the file lives at ~/.lmstudio/mcp.json)
  2. Add the server entry:
{
  "mcpServers": {
    "testmcp": {
      "command": "/opt/homebrew/bin/node",
      "args": ["/path/to/testmcp/packages/mcp-server/dist/index.js"],
      "env": { "GITHUB_TOKEN": "your_github_token_here" }
    }
  }
}

Notes:

  • Use the absolute path to node, not just "node". LM Studio is a GUI app and doesn't inherit your Terminal's PATH, so a bare "node" often resolves to nothing and the server hangs on "Initializing plugin…". Run which node and paste the full path — Homebrew on Apple Silicon is /opt/homebrew/bin/node.
  • Replace /path/to/testmcp with your actual repo path. Run npm run build first so dist/index.js exists.
  • GITHUB_TOKEN is only required for private repos — public repos work unauthenticated (rate-limited).
  • Save the file (Cmd+S) and LM Studio loads the server automatically.

7. Testing it

In an LM Studio chat with Qwen2.5 7B loaded and the TestMCP integration enabled, ask:

Analyze facebook/react#34000 for QA risk.

The model should invoke testmcp_analyze_pr, and TestMCP returns the risk score, impacted functional areas, and factors. This PR is the standing smoke test: it should come back LOW (~12/100) with no functional areas matched — the correct answer for an internal compiler change. If it scores higher or flags an area like ads, a substring-matching regression has crept back in (see §10). Follow up with a request for the full prioritized test plan (testmcp_generate_test_plan).

The demo: same MCP server working in Claude Desktop and in LM Studio with wifi turned off. That's model-agnostic, proven live.

8. Use cases beyond the demo

  • Free development backend — hammer TestMCP with unlimited test calls during development without paying API fees; ship with Claude/GPT for production quality.
  • Sensitive document work — summarize or question anything you don't want leaving your machine.
  • Job search drafting — unlimited, private iteration on resumes and cover letters.
  • Bulk text tasks — classifying, tagging, reformatting many small items; small models excel at high-volume, low-complexity work.
  • Offline fallback — planes, outages, bad wifi.

What local models on 8 GB are not good for: complex reasoning, very long documents, anything where quality matters more than privacy or cost. Use frontier cloud models for those.

10. Case study: the demo found a real bug

The very first local run paid for itself. Analyzing facebook/react#34000 (a React compiler PR), TestMCP scored it MEDIUM 30/100 and flagged the ads functional area — on a PR that has nothing to do with advertising.

Root cause was a two-part substring bug:

  • The glob **/*ima* (meant for Google's IMA ad SDK) matched reanimated-no-memo-arg.js in the changed files.
  • The keyword matcher used a raw prose.includes("ads"), which finds "ads" inside "loads", "uploads", etc.

The fix: word-boundary regex matching for keywords, and segment-boundary globs (**/ima[-._]*, **/*[-._]ima[-._]*) for file patterns — verified with a 14-case regression test before redeploying. Rerunning the same PR: LOW 12/100, no areas matched, which is the correct answer for an internal compiler change with 9 test files.

Lesson: a demo that surfaces a bug you then diagnose, fix, and verify is a better story than a flawless demo — that loop is the QA discipline the tool exists to support. Also: every substring matcher is a false positive waiting to happen; audit classifier patterns for word/segment boundaries early.

11. Capturing the workflow as AI skills (the plan)

Working with an AI assistant on this project raised a good question: should the project get its own skill (a reusable, saved procedure the assistant follows)? The distinction that clarified it:

  • Memory holds facts about the project — architecture, decisions, where things live. Cheap to maintain, always current.
  • A skill captures a repeatable procedure — checklists, conventions, exact steps — worth writing only once the workflow has stabilized.
Rule of thumb: create the skill after the procedure works end-to-end but before you forget the gotchas. Writing it earlier freezes half-formed decisions; writing it later loses the hard-won details.

Two skills planned for this project once the demo runs end-to-end:

  1. testmcp-dev — repo conventions, how to test against LM Studio vs. cloud models, build/release checklist. Trigger: catching myself explaining the same process twice.
  2. testmcp-demo-prep — the exact live-demo runbook: which repo and PR to use, talking points, and the fallback plan if the model or wifi misbehaves mid-interview.

12. Shipping it: repo, PR, and the merge

The fix didn't stay on my laptop. Full loop, done properly:

  1. Dedicated repo. TestMCP started life inside a general workspace repo; I gave it its own public repo (ramonacraft/testmcp). One project, one repo, one clear story for anyone reading it.
  2. Branch → commit → push. Work done on feature/testmcp, committed with a message that explains the why (the demo catch, the root cause, the verification), pushed with -u to track the remote.
  3. Pre-publish secret scan. git grep -i "ghp_" before pushing a public repo. My hits were all ghp_yourtoken placeholders in the README, which is exactly what should be there.
  4. PR with the story. Even solo, the pull request documents bug → root cause → fix → verification where recruiters and collaborators can see it.
Gotcha: if GitHub says "there isn't anything to compare, entirely different commit histories," your branch was pushed to a repo whose main doesn't share your project's history. Don't force-merge unrelated histories. Give the project its own repo.

A product decision made along the way

Cut TestRail and Xray export adapters from the roadmap. TestMCP's YAML test library is the test case management layer — plain files, diffable, living next to the code they test. Bolting on a third-party TCM tool would undercut the whole premise. Jira integration stays, but scoped to bug and task workflow, not test management.

13. Lessons learned

  • Local = ownership. Downloaded weights can't be revoked, metered, or changed under you. That was the whole point.
  • Privacy is structural, not a policy. Data can't leak to a server it's never sent to.
  • Architecture beats configuration. Building on MCP made TestMCP model-agnostic before I ever thought about local models — the protocol keeps the model on the client side.
  • Match tools to the goal. Ollama is great for CLI/API workflows; LM Studio is the move when you need an MCP client with a GUI. I didn't need both.
  • Small hardware forces good habits. 8 GB made me learn quantization, model sizing, and where the quality floor for tool-calling sits (7B).
  • The career story writes itself. "Enterprise-ready" claims are cheap; demoing your product against an air-gapped model with wifi off is proof.
